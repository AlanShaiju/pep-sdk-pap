PHASE 1

Compile Time: Annotation Processor Validation

What happens

On mvn compile, javac discovers the processor via META-INF/services/javax.annotation.processing.Processor. For each @PapEntity class it validates against the catalog and emits one metadata.json per module.

PHASE 2

Startup Time: Bean Creation & Registration

What happens

PapSdkAutoConfiguration creates beans in dependency order: loads metadata into the registry, builds the resilient HTTP client, registers the Hibernate listener, and runs Flyway migrations.

    B1b --> ResolveFail{"Dot-paths resolve?"}
    ResolveFail -->|NO| FailResolve["FAILURE: dot-path fails, app stops"]
    ResolveFail -->|YES| B1c["build Map of Class to PapEntityDescriptor"]
    B1c --> DupCheck{"Duplicate entity class?"}
    DupCheck -->|YES| FailDup["FAILURE: same class in two modules"]
    DupCheck -->|NO| Bean2["2. PapCatalog parse pap-endpoints.json"]
    Bean2 --> CatCheck{"Catalog valid?"}
    CatCheck -->|NO| FailCat["FAILURE: malformed catalog"]
    CatCheck -->|YES| Bean3["3. PapRequestBuilder"]
    Bean3 --> Bean4["4. RestClient with base-url"]
    Bean4 --> UrlCheck{"base-url set?"}

PHASE 3

Request Handling: HTTP POST → PAP Dispatch → Divergence

What happens

A @Transactional method saves three entities. Hibernate events buffer them in order; beforeCommit() dispatches each to the PAP and writes a PENDING row on success; afterCompletion() deletes those rows on commit or flips them to REJECTED_DATA on rollback.

graph TD
    HTTPStart([HTTP POST /resources/ordered]) --> OpenTx["Spring @Transactional opens TX, ROLLBACK-ABLE"]
    OpenTx --> Save1["save Pipeline -> Hibernate POST_INSERT"]
    Save1 --> Listener1["PapEntityListener.onPostInsert()"]
    Listener1 --> ActiveCheck{"TX active?"}
    ActiveCheck -->|NO| FailNoTx["FAILURE 1: no active transaction"]
    ActiveCheck -->|YES| GetBuffer["get/create ChangeBuffer in ThreadLocal"]
    GetBuffer --> Reg["first time: registerSynchronization()"]
    Reg --> Append1["append PapEntityChange Pipeline CREATE SYNC"]
    Append1 --> Save2["save Deployment -> POST_INSERT"]
    Save2 --> Append2["append, buffer = Pipeline, Deployment"]
    Append2 --> Save3["save Monitor -> POST_INSERT"]
    Save3 --> Append3["append, buffer = Pipeline, Deployment, Monitor"]
    Append3 --> Return["controller returns, TX still open"]
    Return --> BeforeCommit["beforeCommit CRITICAL PHASE"]
    BeforeCommit --> Flush["entityManager.flush() runs deferred SQL"]
    Flush --> FlushCheck{"flush OK?"}
    FlushCheck -->|NO| FailFlush["FAILURE 2: constraint violation, rollback"]
    FlushCheck -->|YES| Drain["drain buffer in order"]

    BuildCheck -->|NO| FailBuild["FAILURE 4/5: dot-path null or unknown entity, rollback"]

    Status -->|CB open| FailCb["FAILURE 9: CallNotPermitted, fail-fast"]
    Status -->|2xx| Success1["Pipeline accepted"]
    Fail4xx --> Rollback["beforeCommit throws -> TX rollback, B and C never sent"]
    Fail5xx --> Rollback
    FailTo --> Rollback

    Success1 --> TxId["currentOrNewTransactionId()"]

    ThreePending --> JdbcCommit["Spring issues JDBC COMMIT"]

Fail-fast dispatch

If one entity fails, the rest are never sent and the whole transaction rolls back.

Lazy transaction_id

Drawn from the sequence only on the first PAP success; reused by later entities.

Durable PENDING rows

Written in a REQUIRES_NEW transaction, so they survive a business-TX rollback.

Undetected window

PAP success followed by a failed PENDING insert leaves no trace (Failure 10).

OVERVIEW

Complete System Overview

How the phases connect

Compile-time metadata is bundled into the JAR, read at startup to build the registry and listener, then used at request time to dispatch entity changes to the PAP with divergence tracking.

graph LR
    subgraph Compile["COMPILE TIME"]
        C1["@PapEntity classes"] --> C2["PapAnnotationProcessor validates"]
        C2 --> C3["metadata.json per module"]
    end
    subgraph Startup["STARTUP TIME"]
        S1["MetadataLoader reads all metadata.json"] --> S2["PapEntityRegistry built"]
        S2 --> S3["PapClient + Retry + CircuitBreaker"]
        S3 --> S4["PapEntityListener on Hibernate events"]
        S4 --> S5["Flyway creates STL_PEP_DIVERGENCE"]
    end
    subgraph Request["REQUEST TIME (SYNC)"]
        R1["entity saves -> Hibernate events"] --> R2["ChangeBuffer captures in order"]
        R2 --> R3["beforeCommit dispatches to PAP"]
        R3 --> R4["PENDING rows written per entity"]
        R4 --> R5["afterCompletion resolves"]
        R5 --> R6a["COMMITTED: delete rows"]
        R5 --> R6b["ROLLED_BACK: mark REJECTED_DATA"]
    end
    C3 -.bundled in jar.-> S1
    S5 -.ready.-> R1

Metadata

Compile-time generation avoids runtime reflection on the hot path.

Initialization

Startup pre-builds every dispatch structure once.

Dispatch

Requests reuse those structures for low-latency PAP calls.

Safety

Divergence rows provide crash safety and a manual recovery path.
